Build Guide

What strong connectors include

A real use case with clear actions and outcomes.
API docs or endpoint notes for the service you want to connect.
Explicit auth, scopes, and install settings, with user-owned account connections when possible.
Plain-English copy so users know what the connector does.
Structured outputs that Hollow can route cleanly.
A maintained integration someone actually owns.

Copy Prompt

Generate a connector draft with install metadata, auth setup, and runtime action definitions. 

You are helping me build a Hollow connector.

I need a clean first draft for a connector called [CONNECTOR NAME] that connects Hollow to [PRODUCT OR API].

Assume the developer should be able to connect any HTTP API they want, including REST, GraphQL, XML/SOAP, webhook-based tools, or weird vendor APIs with custom headers.

Context:
- User job to solve: [WHAT THE USER WANTS TO DO]
- API docs or endpoints: [PASTE DOCS OR URL NOTES]
- Auth model: [api_key, bearer, basic, oauth2, webhook, query_param, custom header, signed request, or none]
- Preferred account model: [prefer user-owned account connection / bring your own API key / creator-hosted]
- Important scopes or permissions: [SCOPES]
- Main actions I need: [ACTION 1], [ACTION 2], [ACTION 3]

Return:
1. A short plain-English summary of what the connector does.
2. A Hollow connector JSON payload with:
   - name
   - slug
   - tagline
   - description
   - category
   - visibility
   - status
   - auth_type
   - setup_mode
   - install_copy
   - settings_schema
   - auth_schema
   - scopes
   - actions
   - manifest
3. For each action, include:
   - key
   - name
   - description
   - input_schema
   - output_schema
   - request with method, url, headers, query, body, body_mode, content_type, response_format, response_path, and success_status_codes when needed
4. Include setup_mode, install_copy, settings_schema, and auth_schema when the install experience should feel guided or OAuth-first.
5. Use {{settings.*}} for secrets or saved install values.
6. Use {{input.*}} for runtime values from the user.
7. Include a sample install payload with settings.
8. Include a sample invoke payload for each action.
9. If the API needs custom auth headers, form encoding, XML/text payloads, or non-2xx success semantics, model that explicitly in the request definition.

Important constraints:
- Keep this manifest-driven. Do not invent arbitrary backend code execution.
- Make the copy clear and non-hypey.
- Prefer OAuth or other user-owned account connections when the upstream API supports them.
- Do not assume the developer is paying third-party API costs for every end user unless that is explicitly requested.
- Prefer simple action names and minimal required scopes.
- If the docs are unclear, call that out and make the safest reasonable assumption.
